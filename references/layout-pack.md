# Minimal Terminal Card Layout Pack

These layouts are adapted from infographic layout thinking, but constrained to the approved minimal terminal screenshot style.

## Approved Starting Layouts

### comparison

Use when the source idea is a contrast.

Best for:
- bad vs good
- before vs after
- weak vs strong
- manual vs automated
- option A vs option B

Content shape:

```text
Title
Subtitle
Left heading / Right heading
3-5 paired rows
Bottom rule
```

Notes:
- Keep each row symmetrical.
- Avoid paragraph cells.
- Comparison cards should feel immediately scannable.

### loop

Use when the source idea compounds or repeats.

Best for:
- memory loops
- feedback cycles
- verification cycles
- daily/weekly operating rhythms
- flywheels

Content shape:

```text
Center concept
5-6 surrounding nodes
Directional arrows
Bottom rule
```

Notes:
- Prefer clean orthogonal routing.
- Avoid loose curved arrows unless they look intentionally hand-placed.
- Arrowheads must stop before boxes.

### matrix

Use when the idea is about ownership, mapping, responsibilities, or capabilities.

Best for:
- which agent owns what
- role → responsibility → output
- model selection
- tool capability maps

Content shape:

```text
3 columns max
4-6 rows max
Bottom rule
```

Notes:
- Keep column labels short.
- Use muted text for explanations and bright text for outputs.

### hub-spoke

Use when one central concept coordinates surrounding modules.

Best for:
- command center / central interface
- platform/ecosystem maps
- central interface + capabilities

Content shape:

```text
Center box
4-6 surrounding boxes
Spokes
Short footer label inside diagram
Bottom rule
```

Notes:
- Spokes should stop before boxes.
- Outer panel must not collide with bottom dividers.
- Do not overpopulate spokes.

### agency-map

Use when the source idea is an org chart or multi-agent team.

Best for:
- multi-agent team setup
- AI agency operating model
- user → operator → specialist structure

Content shape:

```text
User/principal
CEO/operator/interface
Specialist layer
Optional analyst/reviewer layer
Delegation arrows + proof return paths
```

Notes:
- Keep delegation and return/proof flows visually distinct.
- Use dashed paths for return/proof flows.
- Do not let arrows cross through boxes.
